# S-Semi 반도체 시료 생산주문관리 시스템 구현 계획

## 시스템 목적

S-Semi의 반도체 시료 생산을 관리하는 콘솔 애플리케이션이다. 운영자는 이 시스템을 통해 시료의 재고 현황을 파악하고, 고객 주문을 접수·승인·거절하며, 생산 라인의 진행 상황을 모니터링하고, 완성된 주문을 출고 처리한다.

생산 라인은 프로그램이 꺼진 동안에도 시간이 계속 흐른다고 간주하며, 프로그램이 재시작되거나 메뉴를 조작할 때 현재 시각을 기준으로 완료된 생산을 자동 반영한다.


## 핵심 도메인

### 시료 (Sample)

반도체 공정에서 생산 가능한 시료의 종류.

| 속성 | 설명 |
|------|------|
| 시료 ID | `S0001` 형식 고유 식별자 |
| 이름 | 시료 명칭 |
| 평균생산시간 | 시료 1개 생산 소요 시간 (0.1 ~ 2.0 min/ea) |
| 수율 | 투입 대비 정상 생산 비율 (0.60~1.00) |
| 현재 재고 | 출하 가능한 완성품 수량 |

### 주문 (Order)

고객이 특정 시료를 요청하는 단위.

| 속성 | 설명 |
|------|------|
| 주문 ID | 자동 부여 (`ORD-YYYYMMDD-NNNN` 형식) |
| 시료 ID | 주문 대상 시료 |
| 고객명 | 주문 고객 |
| 주문 수량 | 요청 수량 |
| 주문 상태 | 아래 상태 전이 참조 |
| 실생산 수량 | 승인 시 계산. 재고 부족분을 채우기 위해 실제 투입할 원자재 수량 |
| 총 생산 시간 | 실생산 수량 × 평균생산시간 (분) |
| 대기열 진입 시각 | 생산 라인 진입 시 기록 |
| 생산 시작 시각 | 실제 생산 시작 시 기록 (대기열 앞순서 도달 시) |
| 예상 완료 시각 | 생산 시작 시각 + 총 생산 시간 |

### 주문 상태 전이

```
[접수]
  reserved
      │
  [승인 결정]
      ├── 승인-├---→ 재고 충분 ───→ confirmed
      │          │                           │
      │          │                        [출고 처리]───→released
      │          │                           │
      │         재고 부족 → producing ─→ confirmed
      │                            
      │                         
      └── 거절 ──────────────────────→ rejected
```

| 상태 | 의미 | 전이 방법 |
|------|------|-----------|
| `reserved` | 주문 접수 완료, 승인 대기 | 주문 생성 시 자동 |
| `confirmed` | 출고 대기 (재고 있음 또는 생산 완료) | 승인(재고 충분) 또는 생산 완료 |
| `producing` | 생산 라인에서 생산 중 또는 대기 | 승인(재고 부족) 시 자동 |
| `release` | 출고 완료 | 운영자 수동 출고 처리 |
| `rejected` | 거부됨 | 운영자 수동 거절 |

> `rejected`는 모니터링에서 제외한다.

### 실생산 수량 계산 (승인 시점에 산출)

재고가 부족할 때만 생산한다. 수율 손실과 10% 공정 오차를 감안해 여유 있게 원자재를 투입한다.

```
부족분        = 주문 수량 - 현재 재고   (음수이면 0, 이 경우 생산 없음)
실생산 수량   = ⌈ 부족분 ÷ (수율 × 0.9) ⌉
총 생산 시간  = 평균생산시간 × 실생산 수량  (분)
예상 완료 시각 = 생산 시작 시각 + 총 생산 시간
```

### 생산 라인

- **FIFO 대기열**: 생산 라인에 먼저 진입한 주문이 먼저 생산된다.
- **단일 처리**: 동시에 한 주문만 생산한다 (capacity = 1).
- 앞 주문이 완료되면 다음 주문의 생산이 자동으로 시작된다.

### 시간 처리 원칙 (Lazy)

- 생산 시작 시각과 예상 완료 시각을 JSON 파일에 기록한다.
- **프로그램 시작 시** 및 **각 메뉴 진입 시마다**, 현재 시각을 기준으로 완료된 `producing` 주문을 `confirmed`로 일괄 갱신한다.
- 이때 대기열의 다음 주문도 순서에 따라 연쇄적으로 처리된다.
- 프로그램이 꺼져 있던 시간도 올바르게 반영된다.

## 주요 기능

### 1. 시료 관리
- 시료 등록 (시료ID, 이름, 평균생산시간, 수율 입력)
- 전체 시료 목록 조회 (재고 포함)
- 시료 검색 (이름 등 속성)

### 2. 시료 주문
- 시료ID, 고객명, 주문수량 입력 → 주문 접수 (`reserved`)
- 주문 ID 자동 부여

### 3. 주문 승인/거절
- `reserved` 주문 목록 표시
- **승인**: 재고 충분 → `confirmed` / 재고 부족 → `producing` (생산 라인 자동 등록)
- **거절**: 즉시 `rejected`

### 4. 모니터링
- 주문 상태별 목록 (`reserved` / `confirmed` / `producing` / `release`)
- 시료별 재고 현황 (여유 / 부족 / 고갈 표시)

### 5. 생산 라인 조회
- 현재 생산 중인 주문과 잔여 시간
- FIFO 대기열 목록 및 순서

### 6. 출고 처리
- `confirmed` 주문 목록에서 선택 → `release` 전환

## 구현 단계

### Phase 1: 프로젝트 기반 구성
**설계 문서**: `docs/design/phase1.md`

참조 저장소의 코드를 이식하고, 빌드 가능한 빈 뼈대를 완성한다.

### Phase 2: 시료·주문 데이터 모델 및 영속성
**설계 문서**: `docs/design/phase2.md`

시료(재고 포함)와 주문(타임스탬프 포함)을 JSON 파일로 읽고 쓰는 기능을 구현한다.

### Phase 3: 비즈니스 서비스 레이어
**설계 문서**: `docs/design/phase3.md`

유효성 검사, 주문 상태 전이 규칙, 실생산 수량 계산 공식을 서비스 계층에 캡슐화한다.

### Phase 4: 생산 라인 시뮬레이터
**설계 문서**: `docs/design/phase4.md`

Lazy 시간 처리와 FIFO 대기열 관리를 구현한다. 시간 의존성을 인터페이스로 추상화해 테스트 가능하게 만든다.

### Phase 5: 콘솔 UI 완성
**설계 문서**: `docs/design/phase5.md`

운영자가 콘솔 메뉴를 통해 모든 기능을 사용할 수 있도록 MVC 기반 UI를 완성한다.

### Phase 6: 통합 검증
**설계 문서**: `docs/design/phase6.md`

전체 시나리오가 실제로 동작하는지 종단 검증한다.

## 핵심 설계 결정

| 결정 사항 | 선택 | 이유 |
|-----------|------|------|
| JSON 라이브러리 | 자체 제작 JsonLib | 외부 의존 없이 DataPersistence 패턴 재사용 |
| 시간 처리 | Lazy (조회 시점 갱신) | 프로그램 종료 중 경과 시간도 정확히 반영, 백그라운드 스레드 불필요 |
| 생산 대기열 | FIFO (단일 라인) | 확정 순서대로 공정하게 처리, 단순 구현 |
| 재고 차감 시점 | 승인 시점 | 승인 즉시 재고를 선점해 이중 할당 방지 |
| 상태 전이 검증 | 서비스 레이어 | 잘못된 전이를 Repository 이전에 차단 |
