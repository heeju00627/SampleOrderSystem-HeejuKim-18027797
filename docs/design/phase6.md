## 검증 시나리오

### 시나리오 1: 재고 충분 — 즉시 출고 (기존 데이터 활용)

```
기존 데이터: ORD-20260508-0001 (S0001, 삼성전자, qty=50, reserved) / S0001 재고=150

1. 모니터링 → 시료별 재고 현황 확인
   - S0001: 재고=150, 예약수요=50, 잔여율=300%, 여유

2. 주문 승인/거절 → ORD-20260508-0001 승인
   - 재고 150 >= 50 → status=confirmed, S0001 재고=100
   - 승인 완료 — 출고 대기(confirmed) 메시지 확인

3. 출고 처리 → ORD-20260508-0001
   - status=released 확인

4. 모니터링: released 목록에 ORD-20260508-0001 확인, S0001 재고=100 확인
```

### 시나리오 2: 재고 부족 — 생산 후 출고

```
기존 데이터: ORD-20260508-0002 (S0002, SK하이닉스, qty=100, reserved) / S0002 재고=80

1. 주문 승인 → ORD-20260508-0002
   - 부족분 = 100 - 80 = 20
   - productionQty = ceil(20 / (0.82 × 0.9)) = ceil(27.1) = 28
   - totalProductionMinutes = 28 × 1.8 = 50.4분
   - status=producing, S0002 재고=0 (80개 선점 차감)
   - 승인 완료 — 생산 등록(producing) + 실생산량·예상완료 표시 확인

2. 생산 라인 조회
   - 현재 생산 중: ORD-20260508-0002, 잔여 시간 표시 확인

3. 재시작 (50.4분 이후)
   - applyLazyUpdates() 자동 호출
   - status=confirmed (재고 변동 없음: 승인 시 이미 선점 차감됨)

4. 출고 처리 → status=released
```

### 시나리오 3: 기존 confirmed 주문 출고 처리

```
기존 데이터: ORD-20260508-0003 (S0003, LG이노텍, qty=200, confirmed)

1. 출고 처리 메뉴 진입 → confirmed 목록에 ORD-20260508-0003 확인
2. 출고 처리 → "출고 처리하시겠습니까?" 확인 프롬프트
3. Y 선택 → status=released, "출고 완료" 메시지 확인
```

### 시나리오 4: 프로그램 재시작 후 생산 상태 복원

```
전제: 시나리오 2에서 ORD-20260508-0002가 producing 상태로 저장됨

1. 프로그램 종료 (estimatedCompletionAt 타임스탬프가 JSON에 저장됨)
2. 50.4분 이상 경과 후 재시작
3. 메뉴 진입 시 applyLazyUpdates() 자동 호출
4. 모니터링: ORD-20260508-0002가 confirmed 상태로 전환됨 확인
```

### 시나리오 5: FIFO 대기열 (기존 S0003 활용)

```
기존 데이터: S0003 (SiC 기판, avgTime=0.6 min/ea, yield=0.95, 재고=0)

1. 주문 A 접수 (S0003, qty=10) → 승인
   - 부족분=10, productionQty=ceil(10/(0.95×0.9))=ceil(11.7)=12
   - totalProductionMinutes = 12 × 0.6 = 7.2분
   - status=producing, 생산 시작

2. 주문 B 접수 (S0003, qty=5) → 승인
   - status=producing (대기, productionStartedAt 비어 있음)

3. 생산 라인 조회: A 생산 중(잔여 ~7분), B 대기 1번 확인

4. A 완료 시각 경과 후 재시작 또는 메뉴 진입
   - A=confirmed, B 생산 시작 확인
```

### 시나리오 6: 시료 검색 및 잘못된 입력 처리

```
6-1. 시료 검색
   - 키워드 "GaN" 검색 → GaN 버퍼(S0001), AlGaN 에피(S0002) 결과 확인
   - 키워드 "없는시료" 검색 → (데이터 없음) 출력 확인

6-2. 잘못된 입력
   - 존재하지 않는 시료ID(S9999)로 주문 접수 → 오류 메시지, 진행 불가
   - confirmed 주문 재승인 시도 → 오류 메시지
   - 숫자 입력 칸에 문자 입력 → 오류 메시지 + 재입력 요청
   - 주문 거절 시 "정말 거절하시겠습니까?" 프롬프트에 N 선택 → 취소 확인
```

### 시나리오 7: 손상된 JSON 파일 안전 처리

```
※ "빈 파일" = 길이 0인 파일 ([]가 아닌 완전히 빈 파일)

- data/samples.json을 "[]" (빈 배열)로 교체 후 시작 → 빈 목록으로 정상 시작
- data/samples.json을 길이 0인 빈 파일로 교체 후 시작
  → JsonLib ParseError 발생 → 앱 종료 또는 오류 메시지 표시
  (길이 0인 파일은 유효한 JSON이 아니므로 파싱 실패가 정상 동작)
```

## 수동 검증 체크리스트

### 기능 완성도

- [ ] 시료 등록 (avgProductionTime 0.1~2.0 범위 유효성 검사 포함)
- [ ] 시료 목록 조회 (재고·상태 컬럼 포함)
- [ ] 시료 검색 (키워드 부분 일치, 빈 결과 처리)
- [ ] 주문 접수 (orderId 자동 부여 ORD-YYYYMMDD-NNNN 형식)
- [ ] 재고 충분 → confirmed 전이 + 재고 차감
- [ ] 재고 부족 → producing 전이 + 실생산량·예상완료 표시
- [ ] 주문 거절 → rejected (모니터링 목록 미표시 확인)
- [ ] 생산 완료 lazy 처리 → confirmed 전이 (재고 변동 없음)
- [ ] FIFO 대기열 순서 유지 확인
- [ ] 출고 처리 → released (확인 프롬프트 포함)
- [ ] 모니터링: 상태별 목록, 재고 여유/부족/고갈 색상 표시
- [ ] 생산 라인: 잔여 시간(Xh Ym 형식), 대기 순서 표시
- [ ] 재시작 후 생산 상태 복원 (타임스탬프 기반 lazy 처리)

### 안정성

- [ ] 빈 목록에서 모든 메뉴 진입 시 크래시 없음
- [ ] 비정상 입력(빈 문자열, 음수, 범위 초과) 처리
- [ ] `[]` 빈 배열 JSON 파일로 시작 시 정상 동작
- [ ] 길이 0 파일로 시작 시 오류 처리 (크래시 없음)
